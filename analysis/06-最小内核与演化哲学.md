# Claude Code 源码解读：最小内核与演化哲学

## 一、"一"是什么

把 Claude Code 的 800+ 个文件、40+ 个工具、80+ 个命令全部剥掉，剩下的那个不可再分的内核是什么？

是一个 **turn**。

```
人说一句话 → AI 想一想 → 可能做点事 → 回一句话
```

这就是全部。整个 Claude Code 就是这一个 turn 的无限展开。

用代码表达，它就是：

```typescript
while (true) {
  const response = yield* callModel(messages)  // AI 想一想
  if (response.hasToolCalls) {
    const results = yield* runTools(response)   // 做点事
    messages.push(results)                       // 把结果喂回去
    continue                                     // 再想一想
  }
  return response                                // 回一句话
}
```

这 10 行代码就是 Claude Code 的"一"。`src/query.ts` 的 1500 行，不过是这 10 行在遇到真实世界的问题后，一层一层长出来的。

## 二、生长的通用原理

这个"一"是怎么长成 800+ 个文件的？不是通过预先设计，而是通过一个反复出现的模式：

> **遇到问题 → 在 turn 的某个环节上插入一个新的关注点 → 这个关注点本身又是一个可以独立生长的"一"**

这个模式有三个关键特征：

### 2.1 插入而非重写

每一次功能扩展，都不是重写 turn 的核心循环，而是在循环的某个"缝隙"里插入一个新的处理步骤。turn 的骨架从未改变。

### 2.2 关注点自身也是一个内核

每个被插入的关注点，都有自己的最小内核，都可以独立生长。工具系统的"一"是 `buildTool({...})`。权限系统的"一"是 `checkPermissions()`。钩子系统的"一"是 `on(event, handler)`。它们各自从自己的"一"出发，长出了自己的复杂性。

### 2.3 问题驱动，不是预见驱动

没有一个功能是"因为将来可能需要"而加的。每一个功能都是因为遇到了一个具体的、真实的问题。这一点从源码的注释中可以清晰地看到——几乎每个复杂的处理逻辑旁边都有一段注释，解释"为什么需要这个"。

## 三、演化路径：从"一"到"万"

下面追溯 Claude Code 从最小内核出发的完整演化路径。每一步都标注了**驱动生长的问题**。

### 第零层：纯对话

```
人 → AI → 人
```

这是起点。一个 REPL，读取用户输入，发给 Claude API，打印响应。没有工具，没有权限，没有压缩。

### 第一层：工具调用

**问题**：AI 只能说话，不能做事。

**插入点**：在 AI 响应之后，检查是否有 tool_use block。如果有，执行工具，把结果喂回去。

```
人 → AI → [工具调用 → 结果 → AI] → 人
         ↑_____循环_____↓
```

这一步引入了 `Tool` 接口。但注意，Tool 的设计从一开始就是自包含的——每个工具自己定义输入 Schema、执行逻辑、提示词。这不是偶然的，这是"一"的分形：**工具本身也是一个最小内核**，它可以独立生长而不影响其他工具。

### 第二层：权限

**问题**：AI 可以执行任意工具了，但用户不信任它。`rm -rf /` 怎么办？

**插入点**：在工具执行之前，插入权限检查。

```
人 → AI → [权限检查 → 工具调用 → 结果 → AI] → 人
```

权限系统的"一"是 `checkPermissions(input) → allow | deny | ask`。从这个"一"出发，它自己长出了：
- 规则匹配（deny > ask > allow）
- AI 分类器（自动模式）
- 用户确认对话框
- 熔断机制（连续拒绝降级）
- 沙箱隔离

每一个都是因为遇到了具体问题才加的。比如熔断机制——是因为发现 AI 在权限不足时会无限重试。

### 第三层：提示词

**问题**：AI 不知道自己是谁，不知道该怎么用工具，不知道什么该做什么不该做。

**插入点**：在 API 调用之前，构建系统提示词。

```
人 → [构建提示词] → AI → [权限 → 工具 → 结果 → AI] → 人
```

提示词系统的"一"是 `getSystemPrompt() → string[]`。从这个"一"出发，它长出了：
- 工具级提示词（每个工具的 prompt.ts）
- CLAUDE.md 配置注入
- 60+ 种上下文附件
- 静态/动态分界线（缓存优化）
- 条件编译（内部/外部版本差异）

### 第四层：上下文管理

**问题**：对话越来越长，超出了上下文窗口。

**插入点**：在 API 调用之前，检查上下文大小，必要时压缩。

```
人 → [构建提示词] → [上下文压缩] → AI → [权限 → 工具 → 结果 → AI] → 人
```

上下文管理的"一"是 `if (tokens > threshold) compress()`。从这个"一"出发，它长出了 5 种压缩策略（Snip → MicroCompact → Collapse → AutoCompact → ReactiveCompact），每种解决不同粒度的问题。

### 第五层：钩子

**问题**：用户想在工具执行前后插入自定义逻辑，但不想改源码。

**插入点**：在 turn 的每个关键节点上，插入事件发射点。

```
人 → [hooks] → [提示词] → [压缩] → AI → [hooks] → [权限] → [工具] → [hooks] → AI → [hooks] → 人
```

钩子系统的"一"是 `on(event, handler)`。从这个"一"出发，它长出了 50+ 种事件类型、三种执行方式（命令/HTTP/回调）、结构化输出 Schema。

### 第六层：多 Agent

**问题**：有些任务太复杂，一个 Agent 处理不了，或者中间结果太多会污染上下文。

**插入点**：工具层新增一个特殊工具——AgentTool，它的"做点事"就是启动另一个 turn 循环。

```
人 → AI → [AgentTool → (人→AI→工具→AI) → 结果] → AI → 人
                        ↑ 这是一个完整的子 turn
```

这是最深刻的分形：**Agent 的实现就是递归调用 turn 本身**。Fork 模式共享父 turn 的上下文，Subagent 模式创建全新的 turn。但底层都是同一个 queryLoop。

### 第七层：MCP

**问题**：内置工具不够用，用户想接入外部工具。

**插入点**：工具注册表新增一个来源——MCP 服务器。

MCP 的"一"是 `ListTools → CallTool`。外部工具通过 MCPTool 包装后，和内置工具在 AI 视角完全一致。这就是为什么工具系统从一开始就设计为接口驱动——它预留了这个生长点。

### 第八层：命令与技能

**问题**：有些操作不需要 AI 参与（如 /clear），有些操作需要 AI 但要限制工具（如 /commit）。

**插入点**：在用户输入和 turn 之间，插入命令路由。

命令系统的"一"是 `if (input.startsWith('/')) dispatch(command)`。技能是命令的一种特殊形式——它构建一个提示词发给 AI，但限制 AI 只能用特定工具。

## 四、分形结构

回头看整个演化路径，会发现一个惊人的结构对称性：

```
系统级：  while(true) { callModel → runTools → continue }
Agent级：  while(true) { callModel → runTools → continue }  ← 同一个循环
工具级：  buildTool({ call, prompt, checkPermissions })      ← 自包含的"一"
权限级：  checkPermissions → allow | deny | ask              ← 自包含的"一"
钩子级：  on(event, handler)                                 ← 自包含的"一"
命令级：  if (startsWith('/')) dispatch(command)              ← 自包含的"一"
```

每一层都有自己的最小内核，每一层都可以独立生长，每一层都不需要知道其他层的存在。这就是为什么 Claude Code 能长到 800+ 个文件而不崩溃——**复杂性被分形结构消化了**。

## 五、通用原理提炼

从 Claude Code 的演化中，可以提炼出一个通用的系统设计原理：

### 原理一：找到不可再分的"一"

任何系统都有一个最小的、不可再分的操作单元。对于 Claude Code，是一个 turn。对于 Git，是一个 commit。对于 Unix，是一个进程。找到这个"一"，就找到了系统的基因。

### 原理二：让"一"可递归

如果"一"可以调用自己，系统就获得了无限的表达能力。Claude Code 的 Agent 就是 turn 调用 turn。Unix 的 fork 就是进程创建进程。递归是复杂性的来源，也是优雅的来源。

### 原理三：通过插入而非重写来生长

永远不要重写核心循环。遇到新问题时，在循环的某个环节上插入一个新的关注点。这个关注点有自己的接口、自己的实现、自己的生长空间。核心循环只需要知道"这里有个插入点"，不需要知道插入了什么。

### 原理四：让每个关注点都是一个独立的"一"

工具是一个"一"（buildTool）。权限是一个"一"（checkPermissions）。钩子是一个"一"（on/emit）。每个关注点都有自己的最小内核，都可以独立演化。这就是分形——整体的结构在每个局部重复出现。

### 原理五：让问题驱动生长，而非预见驱动

不要因为"将来可能需要"而提前设计。等问题真正出现时，在最小内核的某个环节上插入解决方案。这样长出来的系统，每一个部分都有存在的理由，没有多余的复杂性。

## 六、一句话总结

Claude Code 的设计哲学是：

> **从一个 turn 出发，通过在 turn 的缝隙中插入可独立生长的关注点，让问题驱动系统从简单走向复杂，而核心循环始终是那 10 行代码。**

这不是 Claude Code 独有的智慧。Linux 内核从一个任务调度器长出了整个操作系统。React 从一个 `render(vdom)` 长出了整个 UI 框架。Git 从一个内容寻址存储长出了整个版本控制系统。

好的系统设计，都是从"一"开始的。
